# 开发计划

## 1. 开发策略

本项目采用“先闭环、再增强、后打磨”的迭代方式，不走重型过程，也不在课程项目中引入过度复杂的微服务拆分。核心原则如下：

- 优先保证 P0 业务主流程可以演示
- 所有设计先满足课程可交付，再考虑扩展性
- 后端采用模块化单体，AI 能力独立为辅助服务
- 所有迭代都必须留下可运行、可测试的增量成果

## 2. 里程碑安排

| 阶段 | 时间目标 | 核心目标 | 主要产物 |
| --- | --- | --- | --- |
| 阶段 A | 第 1 周 | 完成需求冻结和技术选型 | 需求基线、分工表、仓库结构、接口草案 |
| 阶段 B | 第 2 周 | 完成概要设计和数据库设计 | 模块设计、表结构设计、接口规范、页面清单 |
| 阶段 C | 第 3-4 周 | 实现 P0 功能骨架 | 前后端基础页面和接口、基础联调 |
| 阶段 D | 第 5 周 | 补齐 P1 亮点功能 | 匹配评分、积分、举报、统计 |
| 阶段 E | 第 6 周 | 集成测试和答辩准备 | 测试报告、演示脚本、部署说明 |

## 3. 迭代计划

### 3.1 迭代一：项目骨架与基础能力

目标：让项目能跑起来，并形成稳定的开发基础。

交付内容：

- Monorepo 目录结构完成
- 前端工程初始化
- 后端 FastAPI 初始化
- 基础数据库脚本和环境配置模板
- 登录/认证/上传的基础接口骨架
- 接口文档和字段命名规范

### 3.2 迭代二：主业务闭环

目标：完成最小可用产品版本。

交付内容：

- 失物/招领发布
- 搜索与筛选
- 匹配列表展示
- 认领申请与问答验证
- 交接确认
- 管理员审核界面与接口
- 站内消息基础能力

### 3.3 迭代三：亮点增强

目标：补足课程展示中的“软件工程完整性”和“产品亮点”。

交付内容：

- 综合匹配评分
- 敏感物品脱敏处理
- 信誉积分与积分流水
- 举报处理流程
- 数据统计面板
- 批量录入能力

### 3.4 迭代四：质量打磨

目标：为最终验收和答辩准备稳定版本。

交付内容：

- 核心功能回归测试
- 页面细节修正
- 缺陷修复
- 部署演练
- 演示数据准备
- 文档补齐和汇总

## 4. 每周例会机制

- 周初：确认本周目标、风险、分支规划
- 周中：同步联调阻塞、接口变更、数据库变更
- 周末：验收本周增量成果，更新文档和测试状态

## 5. 交付定义（Definition of Done）

一个功能任务完成必须满足以下条件：

1. 代码已提交到对应功能分支
2. 至少完成一次同伴 review
3. 接口或字段变更已同步文档
4. 关键路径已完成自测
5. 不引入已知阻塞级缺陷

## 6. 沟通与协作约束

- 页面字段以接口文档为准，不允许前后端私下约定未记录字段
- 数据库表结构以文档和 migration 脚本为准，不允许口头修改
- 联调问题统一记录在 issue 或缺陷表中，不在聊天记录中丢失
