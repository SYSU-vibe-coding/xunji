\section{质量目标与测试策略}

\subsection{质量目标}

项目采用核心业务驱动策略，优先验证课程演示所依赖的 P0 闭环，并把安全和数据一致性作为不可由前端绕过的后端约束。具体目标如下：

\begin{enumerate}
  \item \textbf{功能正确性：}核心角色能够按合法顺序完成业务，非法状态转换得到明确拒绝。
  \item \textbf{安全性：}未认证、越权和敏感图片访问受控；默认密钥、调试验证码和内部 AI 接口不能形成生产漏洞。
  \item \textbf{可靠性：}AI 或通知等外部能力失败时，基础业务按设计降级；任务失败可重试且不产生孤立数据。
  \item \textbf{一致性：}认领、交接、积分、通知和匹配状态在事务内保持一致，并通过唯一约束、CAS 或行锁约束竞态。
  \item \textbf{可维护性：}代码满足 Ruff、Mypy、TypeScript 和构建门禁，API、枚举、迁移与文档同步更新。
  \item \textbf{可验收性：}所有「通过」结论均可追溯到用例、代码、CI 日志或人工验收记录。
\end{enumerate}

\subsection{测试数据与角色}

验收环境至少需要两个普通用户、一个后勤或安保角色、一个管理员，以及已认证、未认证和低信誉状态。数据应包含普通物品、证件类敏感物品、含验证问题的招领、可形成高分匹配的失物/招领对，以及拒绝、申诉、关闭和举报所需异常数据。演示短信仅能对显式白名单号码开放，测试结束后必须关闭调试开关。

测试责任按模块划分：QA 维护计划、缺陷与验收记录；前端负责人维护交互与构建测试；后端负责人分别覆盖用户/物品和认领/治理；AI/运维负责人覆盖模型降级、服务鉴权和环境联调。角色分工不能替代交叉评审，状态机和安全修复至少需要对应模块之外的一名成员复核。

\section{质量保证方法}

\subsection{代码与契约保证}

\begin{enumerate}
  \item Python 使用 Ruff 与严格 Mypy，前端使用 vue-tsc；格式、类型和异步常见错误在提交阶段提前发现。
  \item Backend 采用 router--service--repository 分层，路由测试关注鉴权和契约，服务测试关注事务与状态。
  \item 数据库结构以 Alembic 为真相源，迁移测试检查单一 head、关键约束和重复数据处理顺序。
  \item JSON 字段使用 camelCase，数据库字段使用 snake\_case；枚举来自架构文档，变更需同步 API 和前端类型。
  \item AI 返回不是可信业务事实；分数范围、结构、超时、5xx 和不可用图像都由 Backend 校验并降级。
\end{enumerate}

\subsection{安全与隐私保证}

安全回归覆盖随机且一次性验证码、调试白名单、默认密钥拒绝启动、管理员 bootstrap、内部服务 token、CORS 白名单、SSRF、资产所有权、私有 bucket 和短期签名。敏感图片采用 fail closed：检测失败时保持私有，无权用户得到空图，而不是伪造「脱敏 URL」。

现有自动化已经覆盖响应字段、对象存储适配器和角色授权行为；私有 MinIO 匿名访问、签名 URL 浏览器可达性、凭据轮换和生产日志脱敏纳入真实环境持续演练。

\subsection{持续回归策略}

每次变更执行受影响模块测试；认证、对象存储、状态机、迁移和公共客户端属于高影响共享区域，应同时执行 Backend 全量测试、AI 全量测试、两个前端测试与构建。并发问题修复配套增加可重复的并发测试，并与串行主路径共同回归。API、枚举或迁移变化还必须更新文档，并对 OpenAPI 或共享类型做差异检查。
