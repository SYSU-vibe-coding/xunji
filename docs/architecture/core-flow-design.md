# 核心流程设计

> 具体规则（匹配权重、阈值、验证等级、积分）见 `matching-rules.md` 和 `credit-rules.md`。  
> 具体接口字段见 `../api/*.md`。本文档只讲跨模块的"动作顺序"。

## 1. 失物发布与匹配流程

1. 用户进入失物发布页面
2. 填写类别、名称、时间段、地点、描述
3. 上传图片后触发图片上传接口
4. 主后端在同一数据库事务保存失物、图片及 `MATCH`、`CLASSIFY` 两条 `durable_jobs`
5. 事务提交后接口立即返回；`BackgroundTasks` 只唤醒 runner，不承载可靠投递
6. lifespan runner 通过 `FOR UPDATE SKIP LOCKED` 领取任务，调用 AI 生成分类建议和匹配分数
7. 匹配结果与对应站内通知在同一事务写入，任务随后置为 `COMPLETED`
8. 失败任务按指数退避重试；达到上限置为 `FAILED`，进程重启后数据库中的待处理任务仍可继续

## 2. 招领发布与敏感物品处理流程

1. 拾获者填写招领信息并上传图片
2. 主后端保存招领记录、验证问题，并在同一事务写入 `MATCH`、`CLASSIFY`；含图片时另写 `SENSITIVE` 任务
3. `CERT` 或含图片的招领记录发布时先置 `is_sensitive=1`，接口不等待 AI
4. runner 受控并发逐图检测；只有每张图都明确 `isSensitive=false`、`needsReview=false` 且 `degraded=false` 时才将 `is_sensitive` 置 0
5. 超时、空结果、降级结果、需人工复核或任一敏感结果都保持 `is_sensitive=1`；`CERT` 始终为 1
6. runner 异步执行分类与匹配，匹配结果和通知原子提交

## 3. 认领申请流程

1. 失主查看匹配结果列表
2. 选择目标招领物品并发起认领
3. 系统根据物品类型和用户信誉确定验证等级
4. 普通物品进入一级问答验证
5. 高价值物品可进入二级凭证审核
6. 敏感证件类可走快捷通道
7. 审核通过后生成交接安排

## 4. 交接确认流程

1. 双方约定交接方式和时间地点
2. 认领人确认收到物品
3. 拾获者确认已归还
4. 主后端更新认领状态为已交接
5. 同步将招领记录置为 `RETURNED`；若认领来自匹配结果，则将关联失物记录置为 `FOUND`
6. 记录积分流水和交接记录
7. 发送完成通知

## 5. 管理员审核流程

### 5.1 认证审核

1. 用户提交认证申请
2. 管理员查看资料
3. 通过或驳回申请
4. 更新用户认证状态并通知用户

### 5.2 举报处理

1. 用户提交举报
2. 管理员查看举报目标和说明
3. 判定为有效或无效
4. 如有效，执行扣分/警告/封禁等操作
5. 记录处理日志并通知相关用户

## 6. 异常流程处理原则

- 上传失败：提示用户重新上传，不保存半成品状态
- AI 超时：发布已成功；持久化任务退避重试，敏感图片继续 fail closed
- 后台进程退出：`RUNNING` 锁超时后可由新进程回收，`PENDING` 任务直接继续处理
- 认领冲突：优先保持状态一致，后进入人工审核
- 重复提交：通过幂等策略或前端按钮防抖避免重复写入
