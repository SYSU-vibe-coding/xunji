# 需求基线

## 1. 项目目标

"寻迹"是一个面向校园场景的失物招领平台，以**移动端响应式网页**为主要入口，通过"发布 - 匹配 - 验证 - 交接 - 归档"闭环提升失物找回效率，并降低冒领和隐私泄露风险。

项目本轮开发目标不是做成完整商业产品，而是做出一个结构清晰、核心流程完整、可演示、可测试、可部署的课程项目版本。

## 2. 目标用户

| 角色 | 使用目标 | 主要功能 |
| --- | --- | --- |
| 普通用户（学生/教职工） | 发布失物、查看招领、发起认领 | 登录、认证、发布、搜索、匹配、认领、消息 |
| 后勤/安保人员 | 批量登记和代管拾获物品 | 招领发布、批量导入、认领审核、交接确认 |
| 系统管理员 | 审核与运营管理 | 认证审批、内容审核、举报处理、公告、统计 |

## 4. 明确不做的内容

为保证课程项目可按期交付，以下内容暂不列入当前开发范围：

- 与学校统一认证、教务系统、保卫处系统的真实对接
- 真实短信平台、真实支付、真实身份库接入
- 复杂推荐引擎和在线模型训练
- 多租户、分校区独立运营、复杂运营规则
- 完整消息中台、实时聊天、IM 系统

## 5. 范围冻结原则

- P0 功能一旦进入编码阶段，原则上不再新增页面或新增核心字段
- 新增需求必须先判断是否影响数据库、接口、页面原型和测试范围
- 若新增需求会影响里程碑，则默认延期到 P1/P2

## 6. 业务规则基线

### 6.1 发布规则

- 每条失物/招领记录必须包含类别、时间、地点和至少一种描述信息
- 图片非必填，但若上传图片，需支持多图
- 招领信息最多可设置 3 个验证问题

### 6.2 验证规则

- 普通物品默认走一级问答验证
- 敏感证件类物品固定使用三级凭证与人工核验，不得自动快捷通过
- 高价值物品允许追加凭证验证
- 用户信誉低时可以提升验证等级

### 6.3 状态流转规则

- 失物状态：`寻物中 -> 已找回/已关闭`
- 招领状态：`待认领 -> 认领中 -> 已归还/已关闭`
- 认领状态：`待验证 -> 已通过/已拒绝/申诉中 -> 已交接`
- 交接完成后应自动同步关闭关联失物与招领记录

## 7. 非功能基线

| 类别 | 基线要求 |
| --- | --- |
| 性能 | 普通接口响应时间不超过 500ms，匹配计算不超过 3s |
| 安全 | 密码加密存储，敏感图片限制访问，关键行为保留日志 |
| 可用性 | 用户端关键操作不超过 3 步完成 |
| 可维护性 | 前后端分离，接口文档可更新，模块职责清晰 |

## 8. 版本验收口径

课程验收时至少满足：

1. 一个普通用户完成失物发布
2. 一个拾获者完成招领发布
3. 系统生成匹配结果并展示
4. 普通用户发起认领并通过验证
5. 双方确认交接完成
6. 管理员能查看和处理认证/审核数据
