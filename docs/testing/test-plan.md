# 测试计划

## 1. 测试目标

测试目标是验证“寻迹”系统在课程项目范围内是否满足：

- 核心业务流程正确
- 主要角色权限正确
- 关键状态流转正确
- 常见异常输入得到合理处理
- 基本性能和可用性满足展示需要

## 2. 测试范围

### 2.1 功能测试范围

- 登录与身份认证
- 失物/招领发布
- 搜索与筛选
- 匹配结果
- 认领申请与审核
- 交接确认
- 站内消息
- 举报与后台审核
- 积分变更

### 2.2 非功能测试范围

- 接口响应时间抽样测试
- 关键页面可用性检查
- 敏感信息展示检查
- 权限和越权访问检查

## 3. 测试类型

| 测试类型 | 目标 | 工具 |
| --- | --- | --- |
| 单元测试 | 验证工具类、服务逻辑、状态转换 | 后端/AI 服务：`pytest`（`asyncio_mode=auto`） · 前端：`vitest` |
| 接口测试 | 验证接口参数、返回值、权限和错误码 | `pytest` + `httpx.AsyncClient` 模拟请求 |
| 集成测试 | 验证数据库、对象存储、AI 服务协作 | `pytest -m integration`，依赖 `docker compose up -d mysql minio` |
| UI 测试 | 验证页面表单、提示、跳转和状态展示 | `vitest` + Vue Test Utils |
| 静态检查 | lint 与类型 | `ruff check` / `ruff format --check` / `mypy app`（后端和 AI 服务）|
| 回归测试 | 验证修复后未引入新的问题 | 复用已有用例 |
| 验收测试 | 按课程演示流程验证可交付性 | 按 `acceptance-checklist.md` 人工走查 |

所有质量任务通过 `.github/workflows/ci.yml` 在每次 PR 上自动执行，未通过不合并。

## 4. 测试角色分工

| 角色 | 职责 |
| --- | --- |
| 成员 1 | 统筹测试计划、维护缺陷清单、组织验收 |
| 成员 2 | 负责前端交互和页面测试 |
| 成员 3 | 负责用户/物品相关接口测试 |
| 成员 4 | 负责认领/后台/消息相关接口测试 |
| 成员 5 | 负责 AI 服务、部署环境和集成测试 |

## 5. 进入测试前提

- 数据库初始化完成（`uv run alembic upgrade head`）
- 至少一套联调环境可用
- 核心接口文档已冻结
- 测试账号、测试数据准备完成
- 本地或 CI 上 `uv run ruff check .` / `uv run mypy app` / `uv run pytest` 能通过

## 6. 退出测试标准

- P0 核心场景全部通过
- 阻塞级和严重级缺陷已关闭
- 主要接口无明显越权问题
- 演示链路完整可跑通

## 7. 缺陷分级建议

| 等级 | 说明 |
| --- | --- |
| P0 | 阻塞演示或主流程无法完成 |
| P1 | 关键功能异常，但存在绕过方式 |
| P2 | 普通功能问题或提示不准确 |
| P3 | 文案、样式、低优先级体验问题 |

## 8. 测试数据准备建议

至少准备以下测试账号：

- 普通用户账号 2 个
- 后勤/安保账号 1 个
- 管理员账号 1 个
- 已认证用户和未认证用户各 1 个
- 低信誉用户 1 个

## 9. 重点测试风险

- 认领状态流转不一致
- 敏感物品图片泄露
- 多人同时认领同一物品时的状态冲突
- 交接确认后未同步关闭失物/招领记录
