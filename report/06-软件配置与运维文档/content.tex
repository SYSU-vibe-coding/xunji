\section{配置管理}

\subsection{配置原则}

项目遵循「代码、非敏感默认值、敏感配置分离」的原则。Backend 和 AI 服务通过 Pydantic Settings 从进程环境及可选 \path{.env} 读取配置；Compose 从 \path{deploy/docker/.env} 注入环境变量；前端 API 基址在构建阶段通过 \texttt{VITE\_API\_BASE\_URL} 固化。示例配置可以进入仓库，真实密钥、数据库口令和本地覆盖文件不得提交。

根目录 \path{.gitignore} 忽略 \path{.env} 与 \path{.env.*}，只放行 \path{.env.example}；
\path{.dockerignore} 同时排除各层真实环境文件，防止密钥进入 Docker build context。
凭据遵循定期轮换策略；一旦进入 Git 历史、日志或镜像，必须立即在对应平台撤销并更换。

\section{部署与变更操作}

\subsection{部署前检查}

部署者应确认 Docker Compose 或 Podman Compose 可用、端口无冲突、磁盘容量足够，并从示例文件生成本地配置。所有空白密钥必须填入不同的随机值；共享环境禁止使用示例 token、默认管理员密码或通配 CORS。还应确认目标数据库是新库还是历史 \path{schema.sql} 库，后者不能直接 stamp head。

\noindent\textbf{本地完整栈启动示例}\par
\begin{verbatim}
cp deploy/docker/.env.example deploy/docker/.env
# Edit deploy/docker/.env and fill every required secret.
docker compose --env-file deploy/docker/.env \
  -f deploy/docker/docker-compose.yml config --quiet
docker compose --env-file deploy/docker/.env \
  -f deploy/docker/docker-compose.yml up --build -d
docker compose --env-file deploy/docker/.env \
  -f deploy/docker/docker-compose.yml ps
\end{verbatim}

Backend 容器会先执行 \texttt{alembic upgrade head}。若管理员尚不存在，可在首次部署显式设置 \texttt{BOOTSTRAP\_ADMIN\_ENABLED=true}；创建完成并验证登录后立即恢复为 false。演示短信同样只在本地、精确白名单和双重 Debug 开关下启用。

\subsection{部署后核验}

\begin{enumerate}
  \item 检查六服务状态和最近日志，确认 Backend 迁移完成且没有重启循环。
  \item 访问 Backend 与 AI \path{/health}，再实际查询数据库支持的只读接口。
  \item 验证用户端 18080、管理端 18081 以及 Nginx \path{/api/v1} 代理。
  \item 验证 MinIO bucket 没有匿名策略；普通和敏感资产的签名时长及角色权限正确。
  \item 使用演示账号执行登录、发布、搜索、匹配、认领和管理端审核的最小 smoke。
  \item 记录提交 SHA、镜像摘要、Alembic revision、配置项名称摘要和验证结果，禁止记录秘密值。
\end{enumerate}

课程演示完成后如需清理本地数据，可执行 \texttt{docker compose down -v}，但该命令会删除命名卷，属于破坏性重置，不是备份、恢复或正常回滚手段。

\subsection{持续优化：监控与告警}

生产观测可扩展 HTTP 请求数、延迟分位、4xx/5xx 和业务冲突率；数据库连接池、
慢查询和锁等待；MinIO 容量、错误和签名失败；AI 调用延迟、超时、降级率和费用；
durable job 的 PENDING/RUNNING/FAILED 数、重试次数和最老任务年龄，以及容器的
CPU、内存和磁盘指标。

建议设置三级告警：P0 为敏感资产公开、数据库不可用、迁移失败和主流程完全中断；P1 为错误率或任务积压持续超阈值、MinIO 不可用和 AI 大面积超时；P2 为容量趋势、单接口性能和非关键页面异常。告警必须包含环境、版本、requestId、时间范围和处置手册链接，但不得包含密码、OTP、token 或原始证件数据。

\subsection{恢复流程}

\begin{enumerate}
  \item 宣布维护窗口，冻结写入并记录故障版本、数据库 revision 和对象清单。
  \item 在隔离环境创建空 MySQL 与空 MinIO，验证备份文件和密钥可读。
  \item 先恢复数据库和对象，再核对所有 \path{asset://} 引用所对应对象是否存在。
  \item 使用与备份一致的应用版本启动，确认 revision；按批准方案迁移到目标 head。
  \item 验证管理员登录、普通用户读取、敏感资产权限、发布和交接最小路径。
  \item 经负责人确认后切换流量，保留旧环境只读直至观察期结束。
\end{enumerate}

\subsection{应用与数据库回滚}

生产扩展中，应用回滚切换到上一绿色 commit 对应的不可变镜像，禁止重新构建同名
\texttt{:local}。每次发布记录镜像 digest，并保持新旧版本的配置兼容窗口。

数据库默认采用「备份 + 向前修复」。对于只增加兼容字段的迁移，可先部署扩展结构，再部署应用；删除或收紧约束使用 expand--migrate--contract。Alembic downgrade 只在隔离副本验证无数据损失后使用。若迁移已经改写或删除数据，应从部署前备份恢复或编写专用修复脚本，不能盲目 downgrade/stamp。

\section{持续优化与扩展路线}

\begin{enumerate}
  \item \textbf{集成环境：}在 CI 中启动六服务，执行 MySQL/MinIO 集成、健康等待与浏览器 smoke。
  \item \textbf{持续交付：}以 commit SHA 与语义版本标记镜像，增加环境审批、发布后校验与自动回退。
  \item \textbf{安全入口：}引入统一 HTTPS 网关、私有网络、对象访问域名与最小暴露面。
  \item \textbf{备份恢复：}实现离机加密备份、校验、恢复脚本与定期演练，持续校准 RPO/RTO。
  \item \textbf{集中观测：}增加结构化日志、指标、告警、trace、轮转、容量与 durable job 运维面板。
  \item \textbf{容器供应链：}采用非 root、只读文件系统、镜像 digest、SBOM 和漏洞扫描。
  \item \textbf{迁移演练：}在 MySQL 脱敏数据副本上执行 upgrade/downgrade，对 0004 与 0007 进行专项数据核对。
\end{enumerate}

扩展顺序为：凭据轮换、HTTPS/私有网络、MySQL/MinIO 集成与备份恢复优先；
随后建立不可变镜像与审批式持续交付；再扩展 readiness、集中日志、指标告警和任务运维入口；
最后完成非 root、只读文件系统、SBOM、漏洞扫描与容量优化。
