# 部署设计

## 1. 部署目标

课程项目部署设计的核心目标是：

- 本地开发环境能快速启动
- 小组联调环境尽量稳定统一
- 演示环境部署步骤可重复
- 外部依赖尽量少，避免答辩当天故障

## 2. 运行环境划分

| 环境 | 用途 | 特点 |
| --- | --- | --- |
| 本地开发环境 | 单人开发与自测 | 允许使用 mock 数据和本地配置 |
| 联调环境 | 小组集成测试 | 固定数据库、固定对象存储、固定 API 地址 |
| 演示环境 | 答辩展示 | 数据稳定、账号固定、依赖最少 |

## 3. 服务部署清单

### 3.1 必要服务

- 前端用户端构建产物
- 前端管理端构建产物
- FastAPI 主后端（uvicorn）
- MySQL 数据库
- MinIO 对象存储
- AI 服务

### 3.2 可选辅助服务

- Nginx 统一转发前端静态资源和 API
- 日志文件目录或简单日志聚合工具

## 4. 推荐部署方式

### 4.1 本地开发

采用 `docker compose + 本地源码运行`：

- MySQL 使用容器启动
- MinIO 使用容器启动
- AI 服务可本地源码运行或容器运行
- 前后端在开发机本地启动，便于调试

### 4.2 联调环境

- 使用固定主机或云服务器
- 主后端和 AI 服务使用容器部署
- 数据库和对象存储保留固定地址
- 所有前端成员连接同一测试 API

### 4.3 演示环境

- 尽量复用联调环境
- 冻结数据和版本
- 禁止在答辩前最后一天继续改数据库结构

## 5. 配置管理

### 5.1 配置分类

| 配置项 | 示例 |
| --- | --- |
| 数据库配置 | 主机、端口、库名、用户名、密码 |
| 对象存储配置 | Endpoint、Bucket、Key |
| AI 服务配置 | 服务地址、超时、降级开关 |
| 应用配置 | JWT 密钥、文件大小限制、日志级别 |

### 5.2 配置约束

- 环境变量与配置文件分离
- 示例配置进入仓库，真实敏感配置不进入仓库
- 所有成员本地配置命名保持一致

## 6. 降级与容错设计

- AI 服务不可用时，主后端允许跳过 AI 结果并继续完成发布
- 对象存储不可用时，上传流程应明确失败，不写入残缺业务数据
- 通知推送失败时，不应回滚主业务事务，可采用重试或记录失败日志

## 7. 日志与排错建议

- 主后端输出请求日志、错误日志、关键状态变更日志
- AI 服务记录调用耗时和模型处理结果摘要
- 联调环境保留最近一周日志
