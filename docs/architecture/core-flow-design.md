# 核心流程设计

> 具体规则（匹配权重、阈值、验证等级、积分）见 `matching-rules.md` 和 `credit-rules.md`。  
> 具体接口字段见 `../api/*.md`。本文档只讲跨模块的"动作顺序"。

## 1. 失物发布与匹配流程

1. 用户进入失物发布页面
2. 填写类别、名称、时间段、地点、描述
3. 上传图片后触发图片上传接口
4. 主后端保存失物记录和图片记录
5. 若启用 AI，则调用 AI 服务生成类别建议与标签
6. 主后端根据新发布记录触发匹配任务
7. 匹配结果写入 `match_results`
8. 若分数超过阈值，向相关用户发送通知

## 2. 招领发布与敏感物品处理流程

1. 拾获者填写招领信息并上传图片
2. 主后端保存招领记录
3. AI 服务判断是否为敏感证件类物品
4. 若为敏感物品，则生成脱敏展示图或脱敏标签
5. 用户设置验证问题、保管方式和联系方式
6. 主后端触发匹配流程并生成匹配记录

## 3. 认领申请流程

1. 失主查看匹配结果列表
2. 选择目标招领物品并发起认领
3. 系统根据物品类型和用户信誉确定验证等级
4. 普通物品进入一级问答验证
5. 高价值物品可进入二级凭证审核
6. 敏感证件类可走快捷通道
7. 审核通过后生成交接安排

## 4. 交接确认流程

1. 双方约定交接方式和时间地点
2. 认领人确认收到物品
3. 拾获者确认已归还
4. 主后端更新认领状态为已交接
5. 同步关闭失物和招领记录
6. 记录积分流水和交接记录
7. 发送完成通知

## 5. 管理员审核流程

### 5.1 认证审核

1. 用户提交认证申请
2. 管理员查看资料
3. 通过或驳回申请
4. 更新用户认证状态并通知用户

### 5.2 举报处理

1. 用户提交举报
2. 管理员查看举报目标和说明
3. 判定为有效或无效
4. 如有效，执行扣分/警告/封禁等操作
5. 记录处理日志并通知相关用户

## 6. 异常流程处理原则

- 上传失败：提示用户重新上传，不保存半成品状态
- AI 超时：记录日志，返回基础发布成功结果
- 认领冲突：优先保持状态一致，后进入人工审核
- 重复提交：通过幂等策略或前端按钮防抖避免重复写入
