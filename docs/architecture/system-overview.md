# 系统概览

## 1. 系统定位

"寻迹"是面向高校场景的失物招领平台，主要解决当前校园内失物信息分散、匹配效率低和认领安全性不足的问题。系统以**移动端浏览器网页**为主要入口，辅以管理后台和 AI 辅助服务，形成完整业务闭环。

## 2. 建设目标

- 建立统一的校园失物/招领信息库
- 提供可检索、可筛选、可订阅的物品信息入口
- 通过规则和 AI 辅助生成匹配结果
- 通过分级验证和信用约束降低冒领风险
- 支撑管理员完成审核、处罚、公告和统计工作

## 3. 系统上下文

### 3.1 外部参与者

| 参与者 | 与系统关系 |
| --- | --- |
| 普通用户 | 发布失物/招领、搜索信息、发起认领、接收通知 |
| 后勤/安保人员 | 批量录入拾获物品、参与交接与代管 |
| 系统管理员 | 审批认证、审核内容、处理举报、查看统计 |
| 对象存储服务 | 保存图片、凭证、脱敏结果图 |
| AI 服务 | 提供分类、相似度和脱敏能力 |

### 3.2 系统边界

本项目边界限定为课程项目版本，系统不直接对接学校真实统一身份平台，也不承担真实短信、真实物流、真实支付等外部业务。

## 4. 技术方案概览

| 层次 | 技术方向 | 设计原因 |
| --- | --- | --- |
| 用户端 | Vue 3 + Vite + Pinia（响应式网页，移动优先） | 课程项目易开发、浏览器直接访问、无需小程序审核 |
| 管理端 | Vue 3 + Vite + Pinia（PC 网页） | 方便管理员在 PC 端审核和查看统计 |
| 主后端 | Python 3.12 + FastAPI + SQLAlchemy 2.x（async）+ Alembic，uv 管理 | demo 定位下开发迭代快，类型注解 + pydantic 保证接口契约，异步栈与 AI 调用天然契合 |
| 数据层 | MySQL + 对象存储 | 满足结构化数据和图片文件存储 |
| AI 服务 | Python 独立服务（FastAPI） | 便于接入图像和文本模型，与主后端同语言但独立部署和依赖 |

## 7. 当前架构取舍

### 7.1 为什么不拆微服务

五人团队、课程项目、开发周期有限，微服务会显著增加：

- 接口联调成本
- 环境搭建成本
- 配置和部署复杂度
- 排查问题难度

因此本项目采用"**主后端模块化单体（FastAPI） + AI 独立服务**"的折中方案，既保留模块边界，也控制实现成本。

### 7.2 为什么主后端选 Python

- 项目目标是演示软件工程过程的完整性（类型、lint、测试、CI、文档），Python + FastAPI 在这些"过程展示物"上生态成熟
- 与 AI 服务同语言，共享部分工具链，团队学习曲线平缓
- async 栈天然适合"主后端异步调用 AI / 对象存储"的场景

### 7.3 为什么 AI 独立部署

- 模型依赖较重（可能包含 torch、transformers 等），与主后端运行时隔离可以避免镜像膨胀
- 可独立扩缩容和降级，失败不阻塞主业务
- 与主后端同样是 Python，接口协议简单，改造成本低
