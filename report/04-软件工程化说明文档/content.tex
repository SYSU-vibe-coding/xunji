P2 分级：P0 保证登录、发布、搜索、匹配、认领、交接、通知和后台审核闭环；P1 增加

长期未领公告和更强模型。新需求若影响数据库、接口、页面或里程碑，则进入后续
增量迭代，使期末交付始终聚焦已定义的课程主流程。

\section{Git、分支与评审协作}

\subsection{分支和提交约定}

\path{CONTRIBUTING.md} 定义 \term{main} 为可发布分支、\term{develop} 为集成分支，功能和
修复分别使用 \term{feature/<name>}、\term{fix/<name>}。提交采用短 Conventional Commit，
相关时携带 FR 编号，例如：

\begin{verbatim}
feat(FR-02): add lost item publish api
fix(FR-05): correct claim review status transition
refactor(claim): extract handover transaction into service
test(match): cover score fallback path
docs: update database model
\end{verbatim}

每个 PR 应只处理一个功能或缺陷，关联 issue/FR，说明验证记录；UI 变化附截图，API、枚举、
数据库字段或流程变化同步对应文档。CI 对 main/develop 的 push 与 pull request 都运行，
并使用 concurrency group 取消同一引用上的旧运行，减少过时结果占用资源。

\subsection{跨平台文本与忽略规则}

2026 年 7 月 12 日仓库新增 \path{.gitattributes}：所有文本自动识别并统一 LF，Windows
\path{*.bat} 和 \path{*.cmd} 保持 CRLF，图片、PDF、压缩包和字体标记为 binary，避免
行尾归一化破坏二进制文件。\path{.gitignore} 排除 \path{.env}、虚拟环境、\path{node_modules}、
缓存、覆盖率与 LaTeX 中间产物，同时允许 \path{.env.example} 入库。该组合降低跨平台
整文件 diff 和秘密误提交概率。Secret 扫描与凭据轮换属于生产化安全的持续扩展方向。

\section{数据库迁移与结构演进}

数据库演进以 Alembic 为唯一真相源。\path{backend/alembic/env.py} 导入全部 ORM metadata，
支持异步在线迁移和离线 SQL；开发使用 autogenerate 生成初稿，再由评审者核对数据迁移、
索引、唯一约束、默认值和降级路径。常用命令为：

\begin{verbatim}
cd backend
uv run alembic upgrade head
uv run alembic revision --autogenerate -m "add xxx"
uv run alembic downgrade -1
uv run alembic heads
\end{verbatim}

当前迁移从 0001/0002 基础表，经过内容审核字段、match pair 去重、治理审核意见、评分来源、
举报唯一目标，到 0008 持久任务，形成单一线性 head。0004 在约束前重写外部引用，体现
「先迁移数据、再收紧约束」；0007 完成重复举报整理与唯一约束，部署时按规范
在数据库备份上核对历史记录。

历史 \path{backend/sql/schema.sql} 仅保留参考，不再作为部署入口。由旧脚本创建且没有
\term{alembic\_version} 的数据库不能直接 stamp：必须先备份并逐项比较表、列、索引、唯一
约束、默认值和 nullability，结构完全等价后才可一次性 stamp。CI 对迁移脚本的
结构逻辑与单一 head 进行校验；MySQL 容器中的 upgrade/downgrade 属于持续集成扩展项。

\section{结论}

寻迹已经形成与课程规模匹配的工程基础：monorepo 统一交付、需求与枚举文档化、Python
strict 类型和 Ruff、双前端 TypeScript strict、四份依赖锁、线性 Alembic、五 job CI、
分层测试与质量闭环，以及跨平台行尾和环境忽略规则。最近远端 CI 中 Backend
200 项、AI Service 60 项、用户端 17 项和管理端 15 项测试全部通过，双前端构建与
Backend/AI 镜像构建同步通过。

至此，课程要求的「可运行、可测试、可部署、可追溯」工程化目标已形成闭环。
持续优化可进一步扩展真实依赖集成、浏览器 E2E、自动契约、供应链加固与运行观测，
使当前课程成果可平滑演进为更高规模的工程交付体系。
