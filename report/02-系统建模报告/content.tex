\section{建模目标与方法}

\subsection{文档定位}

本报告使用用例图、领域类图、序列图和状态图描述「寻迹」校园失物招领平台，分别回答「谁使用系统」「核心对象如何关联」「一次业务如何协作」和「哪些状态迁移合法」。四种视图共同刻画从发布、匹配、认领到交接结案的核心业务闭环。

模型遵循两个原则：第一，\textbf{视图各有侧重}，避免在同一张图中混合角色、结构、时序和状态信息；第二，\textbf{跨视图语义一致}，参与者权限、对象名称、关联基数、消息顺序与状态迁移采用统一口径。

\section{系统边界与用例模型}

\subsection{参与者定义}

访客负责注册和登录；普通用户在不同业务情境中扮演失主或拾获者，可搜索、发布、匹配、认领并完成交接。STAFF 表示具备批量登记和代管能力的后勤/安保角色；ADMIN 负责审核和治理。AI 服务和对象存储是由系统内部业务流程调用的外部技术参与者，不直接承担用户业务目标。

\subsection{交互阶段与返回语义}

上传与发布阶段先取得稳定图片引用，再提交失物或招领信息。物品服务校验身份、字段和图片归属后保存记录，并立即向用户端返回物品 ID 与业务状态；因此「发布结果返回」和「候选匹配产生」是前后衔接的两个阶段。

候选匹配阶段由物品服务异步触发。匹配服务调用 AI 服务取得可用的分项分数；AI 不可用时采用文本、地点和时间规则计算。只有满足角色、业务状态和评分阈值的候选才进入结果集合，并在用户订阅匹配时生成通知。

结果查看阶段由用户主动查询，系统返回候选得分、主要依据和当前状态。候选涉及的失物或招领状态变化后，查询结果必须反映新的有效性，避免用户依据过期候选发起认领。

\subsection{AI 协作与安全约束}

AI 服务返回内容需要满足约定的数据结构，总分按照可用维度统一归一。图像维度不可用时只采用文本、地点、时间和结构化属性；辅助服务故障不得阻断物品发布。敏感检测不能以不确定结果解除保护，证件类别始终按敏感物品处理。

\subsection{初始状态判定}

状态不是固定从 \term{PENDING} 开始，而是由创建时验证结果直接决定。\term{LEVEL\_1} 问答通过进入 \term{ANSWER\_PASSED}；\term{LEVEL\_2} 问答通过但缺凭证进入 \term{PROOF\_PENDING}，已有凭证则进入 \term{ANSWER\_PASSED}；\term{LEVEL\_3} 缺凭证进入 \term{PROOF\_PENDING}，已有凭证仍需人工，进入 \term{PENDING}。一级/二级问答失败直接保存 \term{REJECTED} 记录并返回统一失败消息。

\subsection{审核、申诉与终止}

拾获者可审核本人招领的活动认领，管理员可以执行管理审核；\term{APPEALING} 只能由管理员复核。二级或三级认领缺少凭证时不能批准。拒绝后若无其他活动认领，found 从 \term{CLAIMING} 恢复 \term{PENDING}，关联 match 从 \term{CLAIMED} 释放为 \term{READ}。用户申诉时必须确认 found 仍可占用、双方账号活动且原 match 仍有效，然后重新把 found 占用为 \term{CLAIMING}。

\term{TERMINATED} 与 \term{REJECTED} 语义不同：前者表示物品关闭、举报下架或账户治理导致流程无法继续，后者表示验证或审核结论不通过。终止活动认领时还必须失效 match、通知双方并写审计日志，避免 found、claim 和 handover 各自停留在互相矛盾的状态。

\section{四视图的模型一致性}

\subsection{角色到状态的闭环}

用例图中的「发起并验证认领」落到类图的 ClaimRequest、ClaimAnswer、FoundItem 和 MatchResult；序列图中的异步匹配为该用例提供可选 \term{matchId} 上下文；状态图再约束该 ClaimRequest 可以执行的审核、申诉和交接动作。失主只有在 match 关联 lost 属于本人时才能以匹配上下文认领，拾获者只有在 found 属于本人时才能审核和以 FINDER 身份确认，管理员只能通过受角色依赖保护的管理接口复核申诉或执行治理。

\subsection{对象到交互的闭环}

序列图的发布阶段对应 LostItem/FoundItem、ItemImage、VerifyQuestion、OperationLog 和 DurableJob；任务执行阶段对应 MatchResult、Notification 和 DurableJob。状态图的结案迁移对应 ClaimRequest、HandoverRecord、FoundItem、可选 LostItem、MatchResult、CreditLog、Notification 与 OperationLog。四类模型对这些交互采用相同对象和状态口径，使业务结果与关联实体同步变化。

\subsection{状态语义的闭环}

LostItem 使用 \term{SEARCHING}、\term{FOUND}、\term{CLOSED}；FoundItem 使用 \term{PENDING}、\term{CLAIMING}、\term{RETURNED}、\term{CLOSED}；MatchResult 使用 \term{NEW}、\term{READ}、\term{CLAIMED}、\term{EXPIRED}。ClaimRequest 使用图 \ref{fig:claim-state} 的八个状态。用例中的操作必须满足状态图守卫，状态迁移也必须由用例图中具有相应职责的参与者触发。

\section{结论}

本报告从行为、结构、交互和状态四个角度定义「寻迹」系统模型：用例图限定角色和系统边界；领域类图说明用户、物品、匹配、认领、交接及配套对象的关系；序列图描述发布与异步匹配的协作顺序；状态图约束认领、审核、申诉、交接和终止过程。四类模型使用一致的角色、对象、基数和状态语义，共同描述校园失物招领的核心业务闭环。
